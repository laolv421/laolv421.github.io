同源与跨域

Posted on 2018-11-09 | In web安全

同源

什么是同源？

在web概念中，同源指的是“三个相同”，域名、协议和端口，这三个相同的话就视为同一个域，本域下的JS脚本只能读写本域下的数据资源，无法访问其它域的资源。

为什么会有同源？

当一个资源从与该资源本身所在的服务器不同的域或端口请求一个资源时，资源会发起一个跨域 HTTP 请求。

vault详解（一）简介

Posted on 2018-11-09 | In 安全管理工具

HashiCorp Vault 的 Slogan 是 A Tool for Managing Secrets，这个口号很好的描述了该产品的定位。大多数软件项目都会不同程度的用到诸如 登录密码/Token/证书/API Key 等私密信息，这些信息用明码保存显然是很不安全的，如果不小心迁入到版本控制系统，还会造成重大的安全问题。

攻击模型

  1. 窃听任意vault中的通信；防止窃听包括客户端连接vault的窃听和对后端存储的窃听；
  2. 篡改静态数据和传输中的数据；任何篡改应该可以被察觉到，并终止消息；
  3. 无需认证和授权直接可以访问控制数据；所有的请求必须由安全策略控制；
  4. 访问控制数据无需担责；如果审计日志启用，在客户端接收到任何秘密之前，客户端的请求和回复必须记录下来；
  5. 关于存储秘密的机密性。离开vault的任何静态数据在存储后端必须加密；
  6. vault支持高可用性（HA）的配置；

Kerberos协议

Posted on 2018-11-08 | In 认证

常见的身份协议包括OAuth2(2010), OpenID Connect(2014), SAML(2002)，SAML2(2005), WS-Trust(2007)和WS-Federation(2003)等，出这些之外，还有一个最古老的身份认证协议Kerberos v4(1988)。本文会介绍Kerberos v5协议，并且介绍Kerberos在现实中的应用。

Kerberos是以Needham-Schroeder symmetric key protocol为出发点，它是现代计算机网络中最广泛部署的身份验证和授权系统。目前Kerberos的最新版本是v5，由Neuman和Kohl于1993年出版，旨在克服v4版本的限制和安全问题，v5版本在RFC4120中定义最终版，Kerberos第5版Generic Security Service Application Programming Interface( GSS-API )机制在RFC4121中定义最终版，这定义了Kerberos实现应该使用的一组标准函数（API库）。

OAuth2协议基础

Posted on 2018-11-05 | In 认证

概述

传统的客户端-服务器认证模型大致为：客户端想要访问服务器的受限（access-restricted）资源的话，服务器需要“资源所有人”的凭证来进行认证。为了让第三方应用访问服务器的受限资源，需要机制来共享“资源所有人”的凭证，但是这样会产生如下的风险：

  1. 第三方应用需要存储“资源所有人”的凭证，通常需要密码；
  2. 尽管密码认证本身存在安全问题，服务器还是要支持密码认证；
  3. 因为没有缺乏一些细粒度的访问控制，导致第三方应用可以随意访问任意的资源；
  4. “资源所有人”无法取消第三方应用的访问权限；
  5. 因为密码保存在第三方应用，极易出现数据泄露的情况；

The OAuth 2.0 Authorization Framework定义在rfc6749中，它主要解决了第三方客户端授权方面的问题，比如可以控制访问范围、访问时间长度，并且可以访问进行撤销等等。

我们可以这样想象，某人开发了一个网站博客，用户必须注册网站才能进行留言，出此之外他开放了一个接口，通过登陆github认证服务器（假设人人都有github账号）就可以留言了。那么为了在他的网站有github账号接口，就需要先去github中注册一个app，给出开发的网址、需要用户哪些权限等等；注册好之后用户就可以通过github账号来留言回复了。
