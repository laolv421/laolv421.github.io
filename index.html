同源与跨域

Posted on 2018-11-09 | In web安全

同源

什么是同源？

在web概念中，同源指的是“三个相同”，域名、协议和端口，这三个相同的话就视为同一个域，本域下的JS脚本只能读写本域下的数据资源，无法访问其它域的资源。

为什么会有同源？

当一个资源从与该资源本身所在的服务器不同的域或端口请求一个资源时，资源会发起一个跨域 HTTP 请求。

vault详解（三）Secrets Engines

Posted on 2018-11-09 | In 安全管理工具

Secrets engines是存储、产生、加密数据的组织，一些secrets engines仅仅存储和读取数据，例如Redis/Memcached；有些连接了其他服务能够产生动态凭证；还有一些自身提供服务，例如totp generation, certificates等。

Secrets backend用来存放（生成）秘密的地方；不同的 backend 类似于不同的虚拟文件系统（tmpfs，udev 等），可以挂载到不同的目录，读写到不同的 backend 下会有各自不同的作用。

vault详解（一）简介

Posted on 2018-11-09 | In 安全管理工具

HashiCorp Vault 的 Slogan 是 A Tool for Managing Secrets，这个口号很好的描述了该产品的定位。大多数软件项目都会不同程度的用到诸如 登录密码/Token/证书/API Key 等私密信息，这些信息用明码保存显然是很不安全的，如果不小心迁入到版本控制系统，还会造成重大的安全问题。

攻击模型

  1. 窃听任意vault中的通信；防止窃听包括客户端连接vault的窃听和对后端存储的窃听；
  2. 篡改静态数据和传输中的数据；任何篡改应该可以被察觉到，并终止消息；
  3. 无需认证和授权直接可以访问控制数据；所有的请求必须由安全策略控制；
  4. 访问控制数据无需担责；如果审计日志启用，在客户端接收到任何秘密之前，客户端的请求和回复必须记录下来；
  5. 关于存储秘密的机密性。离开vault的任何静态数据在存储后端必须加密；
  6. vault支持高可用性（HA）的配置；

Kerberos协议

Posted on 2018-11-08 | In 认证

常见的身份协议包括OAuth2(2010), OpenID Connect(2014), SAML(2002)，SAML2(2005), WS-Trust(2007)和WS-Federation(2003)等，出这些之外，还有一个最古老的身份认证协议Kerberos v4(1988)。本文会介绍Kerberos v5协议，并且介绍Kerberos在现实中的应用。

Kerberos是以Needham-Schroeder symmetric key protocol为出发点，它是现代计算机网络中最广泛部署的身份验证和授权系统。目前Kerberos的最新版本是v5，由Neuman和Kohl于1993年出版，旨在克服v4版本的限制和安全问题，v5版本在RFC4120中定义最终版，Kerberos第5版Generic Security Service Application Programming Interface( GSS-API )机制在RFC4121中定义最终版，这定义了Kerberos实现应该使用的一组标准函数（API库）。

LDAP协议

Posted on 2018-11-06 | In 认证

目录是一个为查询、浏览和搜索而优化的专业分布式数据库，它呈树状结构组织数据，就好象Linux/Unix系统中的文件目录一样。目录数据库和关系数据库不同，它有优异的读性能，但写性能差，并且没有事务处理、回滚等复杂功能，不适于存储修改频繁的数据。所以目录天生是用来查询的，就好象它的名字一样。LDAP是轻量目录访问协议(Lightweight Directory Access Protocol)的缩写，LDAP是从X.500目录访问协议的基础上发展过来的，目前的版本是v3.0。与LDAP一样提供类似的目录服务软件还有ApacheDS、Active Directory、Red Hat Directory Service。

OpenID协议

OpenID协议是一个去中心化的网上身份认证系统。对于支持OpenID的网站，用户不需要记住像用户名和密码这样的传统验证标记。取而代之的是，他们只需要预先在一个作为OpenID身份提供者（identity provider, IdP）的网站上注册。OpenID是去中心化的，任何网站都可以使用OpenID来作为用户登录的一种方式，任何网站也都可以作为OpenID身份提供者。OpenID既解决了问题而又不需要依赖于中心性的网站来确认数字身份，但是目前OpenID协议已经过时，只有很少的网站支持OpenID，前段时间OpenID的重要支持者stackoverflow也宣布不再支持OpenID登录。下图是stackoverflow支持OpenID认证的截图：

  1. 授权码（Authorization Code）
  2. 隐式授权（Implicit Authorization）
  3. 密码凭据（resource owner password credentials）
  4. 客户端凭据模式（client credentials）

OIDC的认证流程有三个：

  1. Authorization code flow
  2. Implicit flow
  3. Hybrid flow
