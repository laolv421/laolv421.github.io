OpenID协议

OAuth2和OIDC的不同模式的应用场景

Posted on 2018-11-05 | In 认证

OAuth2的授权和OIDC的认证流程

之前的文章，我们介绍了OAuth2的授权和OIDC的认证流程，OAuth2的授权有四个，先不包括扩展的授权：

  1. 授权码（Authorization Code）
  2. 隐式授权（Implicit Authorization）
  3. 密码凭据（resource owner password credentials）
  4. 客户端凭据模式（client credentials）

OIDC的认证流程有三个：

  1. Authorization code flow
  2. Implicit flow
  3. Hybrid flow

这篇文章将准确探索何时应该使用这些，会包含一些实例，当然可能引入更多的问题，我们也会尽力地在这里解决。

客户端类型

谈到应用场景，首先先讨论一下客户端的类型，也就是OIDC中的relaying party，对于每种客户端类型，可以使用OAuth2中的那些授权和OIDC中的哪些认证流程。

  1. Web应用程序（Web Application）：拥有后端的Web应用，可以使用OAuth2 Authorization Code Grant(with Confidential Client), OIDC Authorization Code Flow(with Confidential Client)；

  2. 桌面本机应用程序（Desktop Native Application）：OAuth2 Authorization Code Grant(with Public Client and PKCE), OIDC Authorization Code Flow(with Public Client and PKCE)；

  3. 移动本机应用程序（Mobile Native Application）：OAuth2 Authorization Code Grant(with Public Client and PKCE), OIDC Authorization Code Flow(with Public Client and PKCE)；

  4. SPA应用程序（Single Page Application App）：OAuth2 Implicit Grant, OIDC Implicit Flow(OAuth2 Authorization Code Grant or OIDC Authorization Code Flow with Public Client could be used, note commentary below)；

  5. Javascript应用程序（Javascript application）：OAuth2 Implicit Grant, OIDC Implicit Flow(OAuth2 Authorization Code Grant or OIDC Authorization Code Flow with Public Client could be used, note commentary below)；

OAuth2的授权

考虑的方面

当考虑使用哪种OAuth2的授权流程时，需要考虑如下几个方面：

  1. 公开客户端和机密客户端；
  2. 自己可控的APP和第三方控制的APP；
  3. 客户端类型：Web应用, 本机应用, User-Agent-based应用；
  4. 用户身份；

Authorization Code Grant


这通常针对Web应用程序或具有可充当机密客户端的服务器端组件的其他系统（保持客户端机密安全）。End User和User agent看不到OAuth2访问令牌或OAuth2身份验证响应中可能包含的任何其他信息，并允许客户端和最终用户进行身份验证。
Used By
有能保持client secret机密的服务端的Web应用，有着公开客户端和PKCE的本机应用。

Implicit Grant


隐式授权的好处是只需要一次调用IdP，但是问题很明显，就是用户代理现在可以看到访问令牌；此授权也缺乏对客户端进行身份验证的能力，也没有刷新tokens的功能。
Used By
主要用于SPA单页应用程序，还有一些基于javascript和user-agent的应用。
Additional Use
在OAuth2的早期阶段，隐式授权倾向于被推荐用于本机应用程序，尤其是手机本机应用。但是现在，通常建议将授权代码授权（使用公共客户端和PKCE）与本机应用程序一起使用。

Resource Owner Password Credential Grant


与隐式授权一样，此授权也具有仅对授权服务器进行一次调用的好处，它允许一个无法与交互式登录集成的应用程序（例如您使用隐式授权和授权授予的时候）。这使得授权服务器在可以与其交互的客户端类型方面具有很大的灵活性，但它还提供了绕过标准化登录工作流机制的机制，该机制可以强制执行诸如双因素身份验证，强制密码重置和类似的理想身份功能之类的操作。

从集中式身份堆栈的角度来看，绕过这些功能会适得其反并且不合需要;即使身份和访问管理功能不是集中的，这在企业中通常仍然是不受欢迎的。此授权授权支持刷新令牌。
Used By
任何有权访问最终用户凭据的客户端，处理自己的登录工作流程都可以执行这种授权方式。
Additional Uses
建议将此授权与您自己的本机应用程序一起使用，因为可以确保对源代码的完全访问和控制，在大型组织中，这可能不是一个可行的方案。

Client Credential Grant


此授权与OAuth2规范定义的其他授权不同，它仅提供对应用程序（或系统）的身份验证，而不是最终用户。刷新令牌不应与此授权一起使用，但OAuth2规范未明确禁止使用。

Use By
任何可以使客户机密保密的东西，不需要最终用户身份验证，并且需要访问第三方资源适用此授权。

OAuth2协议基础

Posted on 2018-11-05 | In 认证

概述

传统的客户端-服务器认证模型大致为：客户端想要访问服务器的受限（access-restricted）资源的话，服务器需要“资源所有人”的凭证来进行认证。为了让第三方应用访问服务器的受限资源，需要机制来共享“资源所有人”的凭证，但是这样会产生如下的风险：

  1. 第三方应用需要存储“资源所有人”的凭证，通常需要密码；
  2. 尽管密码认证本身存在安全问题，服务器还是要支持密码认证；
  3. 因为没有缺乏一些细粒度的访问控制，导致第三方应用可以随意访问任意的资源；
  4. “资源所有人”无法取消第三方应用的访问权限；
  5. 因为密码保存在第三方应用，极易出现数据泄露的情况；

The OAuth 2.0 Authorization Framework定义在rfc6749中，它主要解决了第三方客户端授权方面的问题，比如可以控制访问范围、访问时间长度，并且可以访问进行撤销等等。

我们可以这样想象，某人开发了一个网站博客，用户必须注册网站才能进行留言，出此之外他开放了一个接口，通过登陆github认证服务器（假设人人都有github账号）就可以留言了。那么为了在他的网站有github账号接口，就需要先去github中注册一个app，给出开发的网址、需要用户哪些权限等等；注册好之后用户就可以通过github账号来留言回复了。

vault详解（五）Storage Backends

Posted on 2018-10-26

Vault主要用于生产环境以管理机密。因此，Vault服务的任何停机时间都会影响下游客户端。Vault旨在支持高可用性部署，以确保机器或流程故障的破坏性极小。

Storage Backends是数据真正落盘的地方（注：到达 Storage Backend 时，数据都已经处于加密状态了，安全并不依赖于Storage backend，实际上它是untrusted的），需要在服务启动时在配置里面指定；这类backend同样有很多种，例如：
（1）inmem 存放在内存中（dev 模式下即是使用 inmen backend 的，一旦重启，数据就丢失了）；

（2）file直接存放在本地文件里；

（3）mysql存放在MySQL数据库里；

（4）consul/etcd 存放在 Key-Value 集群中（高可用）；

官方支持的存储引擎包括如下，更多请看官方文档：

MySQL

特点

Mysql支持HA高可用性。

参数

（1）一般参数有address：MYSQL主机ip；database：数据库名称；table：表名称；tls_ca_file：证书路径；max_parallel：最大并发数；

（2）认证参数有username：用户名；password：用户密码；

（3）HA参数有ha_enabled：true或者false；locak_table：指定存储高可用信息的表，默认这些表以_lock作为后缀的，如果不存在，vault会自动创建；

特点

Zookeeper支持HA高可用性。

（1）一般参数address：Zookeeper集群的ip，用逗号隔开；path：数据要在zookeeper中存储的路径；

Consul

特点

Consul存储后端支持HA高可用性。

参数

（1）一般参数包括address：指定consul的地址，建议是local consul agent的地址，不要直接填consul服务器的地址；path：vault数据在consul中存储位置；scheme：vault和consul通信使用的协议，http或者https；service：指定在consul中注册的服务名称，例如“vault”；session_ttl：指定最小的session TTL；lock_wait_time：lock等待时间；

（3）TLS相关参数包括tls_ca_file、tls_cert_file、tls_key_file、tls_min_version、tls_skip_verify。

vault详解（四）Auth Methods

Posted on 2018-10-26

认证操作

Vault auth backend完成的是认证工作：访问者是谁。有多种可通过挂载添加，默认情况下它们会挂载在 auth/ 下，下面是有关认证的操作：

认证方法

在用户认证的问题上，Vault 同样使用了灵活的插件架构，允许多种认证手段，Vault 将其称为验证方法（Auth method），本节列举了Vault支持下的验证方法。

Tokens

Token认证方法是vault内置的方法，也是客户端认证的核心，其他认证方法最终都是转成了token来认证的。Token认证启动时即被加载，且不能禁用。回忆一下，服务器启动时会输出 Root Token，使用 Root token 登录的用户具有系统最高的访问权限。Token 是可继承的，这个继承包括两方面的含义：
（1）持有Token的用户创建新的Token（Child token），除非特别指定，否则Child token权限和原来的Token相同；

（2）当回收（Revoke）某个Token时，其所创建的Child Token，以及Child Token的Child Token都会被一并删除；

属性

token的属性如下：
（1）ID - 令牌的主ID是随机生成的值

（2）Display Name - 可选地，人类可读的显示名称

（3）Metadata - 用于审计日志记录的元数据

（4）Number of uses - 可选，限制使用次数

（5）Parent ID - 可选地，创建此子令牌的父令牌

（6）Policy - ACL策略的关联列表

（7）Source Path - 生成令牌的路径（例如auth/github/login）

虽然 Token 方法非常简便易用，但主要目的是为了支持自身的运行，安全性并不是非常高。对于真正的用户/机器认证场景，Vault 官方推荐使用其他更加成熟的机制，例如 LDAP，Github，AppRole 等，并且这些方法通常有更好的工具支持。

token分类

vault中的token分为两类，其一为service tokens，另一个为batch tokens，这部分会简单比较这两种token的异同点。

service tokens

Service tokens也成为普通vault tokens，他们支持常见的所有特性，例如renewal, revocation, creating child tokens等。

batch tokens

Batch tokens是被加密过的，携带信息用于vault活动的tokens。

token层级结构

通常，当令牌持有者创建新令牌时，这些令牌将被创建为原始令牌的子节点; 他们创造的令牌将是他们的子令牌；当父令牌被撤销时，其所有子令牌及其所有租约也被撤销。这确保了用户不能通过简单地生成永不停止的子令牌树来逃避撤销。

orphan令牌（很少使用）

通常不需要此行为，具有适当访问权限的用户可以创建 orphan令牌。这些令牌没有父级 - 它们是自己的令牌树的根。创建orphan方式如下：
（1）通过auth/token/create-orphan端点

（2）具有sudo功能或root策略，设置orphan参数为true时，访问auth/token/create路径可以创建；

（3）通过token store创建；

（4）使用任何其他（Non-token）身份验证方法登录；

Token Accessor

Token Accessor（令牌访问器），创建令牌时，还会创建并返回Token Accessor；它是一个值，用作对令牌的引用，只能用于执行有限的操作：
（1）查找令牌的属性（不包括实际的令牌ID）；

（2）在路径上查找令牌的功能；

（3）撤消令牌；

PGP、GPG和Keybase

概念

PGP全称是Pretty Good Privacy，是一个基于RSA公钥加密体系的邮件加密软件。PGP主要的特点是它可以在一个非安全的网络环境下，使得从未谋面的人取得信任。相比PKI，PGP根本不需要HTTPS，也不需要CA，仅需要一个可信赖的密钥托管服务器（基于LDAP的服务器）。由于PGP被广泛应用，最终形成一个开放的标准—OpenPGP。

GPG全称是GNU Privacy Guard，是OpenPGP标准（RFC 4880）的独立实现，通过GPG可以对密钥进行管理、对文件和电子邮件进行加密/解密和数字签名等。GPG由于源代码开放，因此很难隐藏后门，比PGP等商业软件更加安全。

为什么使用PGP

为什么要在vault中使用PGP，主要因为早期初始化启动vault后会产生unseal的明文显示的keys，这个就违背了vault的安全初衷，而且也使得分发keys变得困难。之后，vault在0.3版本之后引入了可选择地使用PGP keys进行初始化操作。在这种模式下，当vault初始化后产生unseal keys后，用户的PGP的公钥会立刻加密这些keys。

租期、更新和撤销

对于每个动态的secret和认证token，vault都会创建一个lease（租期），其中metadata（元数据）包括time duration、renewability等。撤销有三种方式，API、CLI和vault自动撤销过期的租期。

vault read命令请求secret，vault总会返回一个lease ID，通过vault renew和vault revoke操作lease ID去管理secret的租期。
