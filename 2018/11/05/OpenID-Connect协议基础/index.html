OpenID Connect协议基础

Posted on 2018-11-05 | In 认证 |

OpenID协议

OpenID协议是一个去中心化的网上身份认证系统。对于支持OpenID的网站，用户不需要记住像用户名和密码这样的传统验证标记。取而代之的是，他们只需要预先在一个作为OpenID身份提供者（identity provider, IdP）的网站上注册。OpenID是去中心化的，任何网站都可以使用OpenID来作为用户登录的一种方式，任何网站也都可以作为OpenID身份提供者。OpenID既解决了问题而又不需要依赖于中心性的网站来确认数字身份，但是目前OpenID协议已经过时，只有很少的网站支持OpenID，前段时间OpenID的重要支持者stackoverflow也宣布不再支持OpenID登录。下图是stackoverflow支持OpenID认证的截图：

背景

用户登录不同的网站需要不同的账号密码，对于用户来说，使用不同的密码并且定期更换时不可能的，维护密码是一大问题。OpenID作为一个无密码登录协议，它的出现很好地解决了这个难题。OpenID只需要使用一套账号密码系统，用这套账号密码可以登录所有支持OpenID协议的网站。

OpenID的出现是在2005年，随即很多公司和组织表示支持OpenID，包括Google、Facebook、Yahoo、Microsoft等等，但是现在由于OAuth的出现导致OpenID的落寞，现在OAuth和OIDC变得愈加流行。

流程

获得OpenID

要想获得OpenID，就要去OpenID Provider注册，OpenID可以是URL或者标识符，例如 http://laolv.openid.org.cn ，任何支持OpenID的网站，用户都可以用此来认证登录。

登录

当用户在网站输入OpenID登录，用户被重定向到OpenID Provider，需要在OpenID Provider认证用户自己的身份，一般采用密码方式。认证成功之后，用户又被重定向回网站。

OpenID Connect协议

OpenID Connect简称OIDC，2014年公布的协议，吸取了SAML、OpenID的经验，并在OAuth2基础上实现身份认证功能。它调用采用API-friendly的RESTful HTTP API形式，同时OpenID Connect实现了可选择的签名和加密算法，并使用JSON格式来操作。现在OpenID Connect已经成为了单点登录和互联网身份认证提供方法的标准。

背景

其他文章介绍OAuth2可以授权第三方通过例如Google账号来访问受保护的信息，但是并不提供账号的信息给第三方，第三方此时是对用户的情况一无所知的。回忆一下，OAuth2的流程是仅能得到access token这类的东西，但是如果第三方需要更多用户的信息，OAuth2是做不到这个的，OIDC此时可以很好地处理这个问题。

特点

OpenID Connect主要有三个方面的亮点：

  1. 易于使用的身份令牌；用户的身份是用JWT（Json Web Token）协议来加密的，这个令牌也被叫做ID token；
  2. 基于OAuth2.0协议；OAuth2.0同时支持web应用和本机应用（native/mobile apps）；
  3. 简单性；集成基本的应用非常容易，同时提供了适应多种场景的不同安全方法选择；

名词解释

主要有RP、OP和EU名词，解释如下：

  1. Relying Party(RP)：依赖方，通常是第三方应用程序(客户端)，例如博客；
  2. OpenID Provider(OP)：OpenID提供方，通常是一个 OpenID 认证服务器（例如开源的Hydra、WSO2），它能为依赖方提供断言以证实用户拥有某个标识；
  3. End-User(EU)：终端用户，指持有账号的人，例如github用户；
  4. ID Token：JWT格式的数据，格式大致为header+body+signature，body中包含EU身份认证的信息，由OP生成；
  5. OpenID Provider Endpoint：OpenID提供方中的端点，主要包括Authorisation endpoint、Token endpoint、UserInfo Endpoint和Optional Endpoint；

不仅仅在基本的登录过程可以使用，ID token还有如下使用场景：

  1. stateless sessions；ID token放入浏览器可以实现一个比较轻型的无状态会话，这就消除了在服务器段存储（包括内存和本地硬盘）session的负担，因为会话可以通过ID token来验证；

  2. 向第三方传递身份；当需要验证身份的时候，可以向其他应用组件或者后端服务传递ID token；

  3. token exchange；ID token可以换取access key，在分布式和企业应用也可以用到；

Authorization code flow

  1. 所有的Token都是通过OP的Token EndPoint来发放的，构建一个OIDC的Authentication Request需要提供如下的参数：

    • response_type：必选，值为code，表示认证流程为授权码流程；
    • scope：必选。OIDC的请求必须包含值为”openid”的scope的参数；
    • client_id：必选。RP在OP中注册的客户端识别码；
    • redirect_uri：必选。RP回调地址；
    • state：推荐。同OAuth2，防止CSRF。
